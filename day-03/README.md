# 📅 Day 03 – Python Dependency Management with `uv` and Lockfiles

## 🗓️ Date: 15-05-2026

## 🎯 Today's Goal

Correct a broken `requirements.in` specification and compile it into a **pinned lockfile** using `uv` — ensuring fully reproducible Python environments across machines.

---

## 📚 Concepts Learned

### What is `uv`?
`uv` is an extremely fast Python package installer and resolver (written in Rust) that replaces `pip` and `pip-tools`. In MLOps it is used to:

- Resolve dependencies **deterministically**
- Compile high-level specs (`requirements.in`) into **pinned lockfiles** (`requirements.txt`)
- Ensure every machine gets the **exact same package versions**

### What is `requirements.in`?
A high-level, human-maintained file that lists only the **top-level packages** your project needs — without pinned versions or transitive dependencies.

```
scikit-learn>=1.3.0
mlflow>=2.0.0
pandas>=2.0.0
numpy>=1.24.0
```

### What is `requirements.txt` (lockfile)?
A machine-generated, fully **pinned** file that includes exact versions of every package — including transitive dependencies resolved by `uv`. This is what gets committed to the repo for reproducibility.

### Why Lockfiles Matter in MLOps?
- Guarantees **identical environments** across dev, staging, and production
- Prevents the "works on my machine" problem
- Makes CI/CD pipelines and Docker builds **deterministic**
- Critical for **model reproducibility** — same code + same deps = same results

## 💻 Commands Used

### 1. Navigate to the Project Directory

```bash
cd /root/code/fraud-detection
```

---

### 2. Review the Existing `requirements.in`

```bash
cat requirements.in
```

> Inspect what packages and version constraints are currently listed. Look for:
> - Missing packages from the required four
> - Wrong package names
> - Invalid or unsatisfiable version constraints

---

### 3. Edit and Correct `requirements.in`

```bash
nano requirements.in
# OR
vi requirements.in
```

The corrected `requirements.in` must contain **exactly these four packages**:

```
scikit-learn>=1.3.0
mlflow>=2.0.0
pandas>=2.0.0
numpy>=1.24.0
```

> ✅ Rules followed:
> - Exactly 4 top-level packages: `scikit-learn`, `mlflow`, `pandas`, `numpy`
> - Every package has a version constraint `uv` can satisfy against PyPI
> - No extra packages, no typos, no invalid constraints

---

### 4. Compile `requirements.in` into a Pinned Lockfile

```bash
uv pip compile requirements.in -o requirements.txt
```

> | Flag | Purpose |
> |------|---------|
> | `requirements.in` | Input: high-level spec file |
> | `-o requirements.txt` | Output: pinned lockfile |

`uv` resolves all transitive dependencies and pins every package to an exact version using `==`.

---

### 5. Verify the Output

```bash
cat requirements.txt
```

> Check that:
> - All 4 top-level packages appear with `==` pinned versions
> - Transitive dependencies (e.g. `scipy`, `six`, `smmap`, `zipp`) are also listed
> - Comments show which top-level package pulled in each transitive dep

---

## 🔁 Workflow Summary

```
cat requirements.in              # Review existing spec
        ↓
nano requirements.in             # Fix packages & version constraints
        ↓
uv pip compile requirements.in -o requirements.txt   # Compile lockfile
        ↓
cat requirements.txt             # Verify pinned output
```

## ✅ Key Takeaways

- `requirements.in` = what **humans write** (top-level only, loose constraints)
- `requirements.txt` = what **machines generate** (fully pinned, includes all transitive deps)
- Always commit **both files** to version control
- `uv pip compile` is deterministic — same input always produces same output
- Pinned lockfiles are essential for **reproducible ML pipelines and Docker images**
- `uv` is significantly faster than `pip-tools` for the same workflow
